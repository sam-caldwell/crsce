# Observability (O11y)

This project includes a unified, lightweight observability system that records:

- Counters: monotonically increasing counts keyed by name and optional tags (each update tracked with `ts_ms`).
- Metrics: latest values keyed by name and optional tags; supports `int64_t`, `uint64_t`, `uint32_t`, `uint16_t`, `double`, `float`, `bool` (each upsert tracked with `ts_ms`).
- Events: point-in-time occurrences with a timestamp and optional tags; may carry an optional string `message`.

O11y writes JSON Lines (JSONL) in the background with minimal overhead. Use it to monitor CLI runs, internal phases, and
test runners without blocking hot paths.

## Design Overview

- Single `O11y` class manages all counters, metrics, and events.
- Fire-and-forget async APIs enqueue work to a background scheduler thread (`std::jthread`).
- Periodic flush (default 250 ms) writes:
    - Events → `CRSCE_EVENTS_PATH` (default `build/events.jsonl`), one JSON object per event.
    - Summary snapshot (counters + metrics) → `CRSCE_METRICS_PATH` (default `build/metrics.jsonl`).
- Clean shutdown flush: best-effort flush on process exit.

## Quick Start (C++)

```cpp
#include "common/O11y/O11y.h"

using crsce::o11y::O11y;

int main() {
  // Optional: start thread early
  O11y::instance().start();

  // Increment counter (async)
  O11y::instance().counter("files_processed");

  // Metrics (async overloads)
  O11y::instance().metric("bytes_total", static_cast<std::int64_t>(1024));
  O11y::instance().metric("ratio", 0.875);     // double
  O11y::instance().metric("ok", true);         // bool
  O11y::instance().metric("u64_example", static_cast<std::uint64_t>(42));

  // Events (async)
  O11y::instance().event("pipeline_begin", {{"stage","ingest"}});
  O11y::instance().event("note", std::string("hello world"));
}
```

All async calls are non-blocking and trigger lazy startup of the background thread on first use.

## File Outputs and Formats

### Events JSONL (`CRSCE_EVENTS_PATH`, default `build/events.jsonl`)

Each event is one JSON object per line. Example:

```json
{
    "ts_ms": 1700000000000,
    "name": "pipeline_begin",
    "tags": {
        "stage": "ingest",
        "file": "a.bin"
    }
}
{
    "ts_ms": 1700000000500,
    "name": "error",
    "message": "I/O transient",
    "tags": {
        "type": "io",
        "code": "ENOENT"
    }
}
```

### Metrics JSONL (`CRSCE_METRICS_PATH`, default `build/metrics.jsonl`)

O11y periodically writes a “summary” snapshot containing all current counters and metrics:

```json
{
    "ts_ms": 1700000001000,
    "name": "summary",
    "counters": [
        {
            "name": "files_processed",
            "tags": "",
            "count": 3,
            "ts_ms": 1700000000900
        }
    ],
    "metrics": [
        {
            "name": "bytes_total",
            "tags": "",
            "t": "i64",
            "value": 2048,
            "ts_ms": 1700000000850
        },
        {
            "name": "ratio",
            "tags": "",
            "t": "f64",
            "value": 0.875000,
            "ts_ms": 1700000000855
        },
        {
            "name": "mode",
            "tags": "key=val|stage=ingest",
            "t": "str",
            "value": "fast"
        },
        {
            "name": "ok",
            "tags": "",
            "t": "bool",
            "value": true,
            "ts_ms": 1700000000860
        }
    ]
}
```

Notes:

- Events include tags as a JSON object for readability.
- Counters/metrics use a canonicalized tag string (sorted `key=value` pairs joined by `|`) for compactness.

## Environment Variables

- `CRSCE_O11Y_FLUSH_MS`: background flush interval in milliseconds (default `250`). Valid range `[10, 60000]`.
- `CRSCE_EVENTS_PATH`: output path for events JSONL (default `build/events.jsonl`).
- `CRSCE_METRICS_PATH`: output path for summary JSONL (default `build/metrics.jsonl`).
- `CRSCE_METRICS_FLUSH`: when set (non-zero), O11y performs an immediate flush after each update to improve determinism for tests. Use sparingly in production as it increases I/O.

## Tags

- Tags are passed as `std::vector<std::pair<std::string,std::string>>` (alias `O11y::Tags`).
- Order does not matter; O11y sorts tags by key for canonicalization.

Example:

```cpp
O11y::instance().metric("throughput", static_cast<std::int64_t>(123), {{"host","A"},{"zone","us"}});
O11y::instance().event("chunk_done", {{"id","42"}});
```

## Lifecycle & Threading

- `O11y::instance()` provides a single process-wide instance.
- `start()` eagerly starts the background worker; any API call will also start it if needed.
- The worker wakes on demand and on a periodic timer to flush.
- On process teardown, O11y requests the worker to stop and performs a best-effort final flush.

## Best Practices

- All O11y APIs are asynchronous and non-blocking by design; no `_async` suffix is used.
- Use events for lifecycle and discrete occurrences; metrics for gauges; counters for totals.
- For CI/tests needing immediate visibility, set `CRSCE_METRICS_FLUSH=1` and/or reduce `CRSCE_O11Y_FLUSH_MS`.
- Add tags for disambiguation (e.g., per-runner, per-file) to simplify log queries.

## Troubleshooting

- “No lines appear”: check `CRSCE_EVENTS_PATH`/`CRSCE_METRICS_PATH` and that the process makes async calls (which start
  the worker). For tests, consider `CRSCE_METRICS_FLUSH=1`.
- “High I/O”: increase `CRSCE_O11Y_FLUSH_MS` or remove `CRSCE_METRICS_FLUSH` in production.

## Migration Notes

- Legacy free-function emitters for metrics/counters/events have been removed.
- All call sites should use the `O11y` class APIs instead.
